# Gemini 코딩 도우미 - 프롬프트 구조 방식 비교

동일한 규칙 세트를 시스템 프롬프트에 구조화하는 방식이 모델 응답에 미치는 영향을 비교하는 실험입니다. `gemini-3.1-flash-lite-preview` 모델을 사용하며, 두 스크립트에 적용된 규칙의 내용은 동일하고 **구조화 방식만 다릅니다.**

---

## mainA.py - XML 태그 + 단순 나열 방식

모든 규칙을 `<role>`, `<rules>`, `<output_format>` XML 태그로 감싸고, 규칙은 카테고리 구분 없이 `-` 기호로 평탄하게 나열합니다. 태그 기반 구조가 모델의 규칙 준수율에 미치는 영향을 확인합니다.

```xml
<role>당신은 코딩 도우미입니다</role>
<rules>
  - 간결하게 말해
  - 읽지 않은 코드를 수정하지 마
  - 위험한 명령은 확인해
  - ...
</rules>
<output_format>마크다운으로 출력</output_format>
```

---

## mainB.py - 마크다운 헤더 + 상황별 카테고리 분류 방식

태그 없이 마크다운 헤더(`#`)로 규칙을 상황별 카테고리(`코드 수정 및 작성`, `위험 행동 통제`, `오류 대처 방안` 등)로 분류합니다. 규칙을 맥락에 따라 묶었을 때 모델이 상황에 맞는 규칙을 더 잘 적용하는지 확인합니다.

```markdown
# 코드 수정 및 작성
- 읽지 않은 코드는 함부로 수정하지 마세요.
- 보안 취약점을 도입하지 않도록 코드를 작성하세요.

# 위험 행동 통제
- rm -rf와 같이 복구 불가능한 삭제 명령은 반드시 사용자에게 먼저 확인하세요.
- git push --force 명령은 절대로 사용자의 명시적 승인 없이 실행하지 마세요.

# 오류 대처 방안
- 에러가 발생하면 발생한 원인과 해결 방법 1가지만 요약해서 제안하세요.
...
```

## 설치 및 실행

**의존성 설치**
```bash
uv add google-generativeai python-dotenv
```

**환경 변수 설정** - 프로젝트 루트에 `.env` 파일 생성
```env
GEMINI_API_KEY="your_api_key_here"
```

**실행**
```bash
# A 방식 (XML 태그 + 단순 나열)
uv run mainA.py

# B 방식 (마크다운 헤더 + 카테고리 분류)
uv run mainB.py
```

`질문:` 프롬프트에 코딩 관련 질문을 입력합니다. `exit` 또는 `quit`으로 종료합니다.
각 응답 하단에 응답 시간(초), 입력/출력/합계 토큰 수가 출력됩니다.
